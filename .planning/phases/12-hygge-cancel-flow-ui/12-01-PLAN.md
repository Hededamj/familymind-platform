Create the directory `app/dashboard/subscription/cancel/` (and the file) and write `actions.ts` containing three server actions. This file is the ONLY source the Plan 12-02/12-03 client components call.

File must begin with `'use server'` on line 1.

Imports:
```typescript
'use server'

import { requireAuth } from '@/lib/auth'
import { prisma } from '@/lib/prisma'
import { cancelSubscription } from '@/lib/services/cancellation.service'
import { resolveEligibleOffer, acceptOffer, type EligibleOffer } from '@/lib/services/retention.service'
import type { RetentionOfferType } from '@prisma/client'
```

Implement three exported async functions with exact signatures:

1. `submitSurveyAndResolveOffer(input: { entitlementId: string; reasonSlugs: string[]; feedback: string | null; primaryReasonSlug: string }): Promise<{ surveyId: string; offerType: RetentionOfferType | null; offerId: string | null; offerConfig: EligibleOffer | null }>`
   - Call `requireAuth()` → get `user.id`.
   - IDOR: `prisma.entitlement.findFirst({ where: { id: entitlementId, userId: user.id, status: 'ACTIVE' } })` — throw `Error('Intet aktivt abonnement fundet')` if null.
   - Resolve primary reason row: `prisma.cancellationReason.findUnique({ where: { slug: primaryReasonSlug } })` → throw if null.
   - Resolve all tag IDs: `prisma.cancellationReason.findMany({ where: { slug: { in: reasonSlugs } }, select: { id: true } })`.
   - Upsert survey using the compound unique `userId_entitlementId`:
     ```typescript
     const survey = await prisma.cancellationSurvey.upsert({
       where: { userId_entitlementId: { userId: user.id, entitlementId } },
       update: {
         primaryReasonId: primary.id,
         feedback: feedback ?? null,
         tags: { deleteMany: {}, create: tagIds.map((id) => ({ reasonId: id })) },
       },
       create: {
         userId: user.id,
         entitlementId,
         primaryReasonId: primary.id,
         feedback: feedback ?? null,
         tags: { create: tagIds.map((id) => ({ reasonId: id })) },
       },
     })
     ```
     If the nested `tags` relation name differs (check schema — it uses the CancellationSurveyTag join model), use the correct relation name; if unsure, first do the upsert without tags, then `prisma.cancellationSurveyTag.deleteMany({ where: { surveyId: survey.id } })` followed by `createMany` with `{ surveyId, reasonId }` rows. Pick whichever compiles clean.
   - Call `const offer = await resolveEligibleOffer(user.id, reasonSlugs)`.
   - Return `{ surveyId: survey.id, offerType: offer?.offerType ?? null, offerId: offer?.id ?? null, offerConfig: offer }`.

2. `acceptOfferAction(input: { surveyId: string; offerId: string; entitlementId: string; pauseMonths?: 1 | 2 | 3 }): Promise<{ acceptanceId: string; expiresAt: Date | null; cancelReversed: boolean }>`
   - Call `requireAuth()`.
   - If `pauseMonths` is provided, first override the offer's pauseMonths by patching the offer row via a passthrough: `acceptOffer()` reads pauseMonths from the offer, so update: `await prisma.retentionOffer.update({ where: { id: offerId }, data: { pauseMonths } })` ONLY IF pauseMonths differs and the offer is of type PAUSE. This is a pragmatic workaround — document with a comment `// TODO(phase-13): plumb pauseMonths through acceptOffer input instead of mutating offer row`.
   - Call `const result = await acceptOffer({ userId: user.id, entitlementId: input.entitlementId, surveyId: input.surveyId, offerId: input.offerId })`.
   - Return `{ acceptanceId: result.acceptanceId, expiresAt: result.expiresAt, cancelReversed: result.cancelReversed }`.

3. `confirmCancelAction(input: { entitlementId: string }): Promise<{ cancelAtPeriodEnd: boolean; currentPeriodEnd: Date }>`
   - Call `requireAuth()`.
   - **Survey-gate workaround** (OFF-DATA-03 requires a survey exists): upsert a minimal survey first.
     ```typescript
     await prisma.cancellationSurvey.upsert({
       where: { userId_entitlementId: { userId: user.id, entitlementId: input.entitlementId } },
       update: {},
       create: { userId: user.id, entitlementId: input.entitlementId },
     })
     ```
     This is a no-op if a full survey already exists from Step 2 (Step 2 sets primaryReasonId; upsert update is empty so those fields are preserved). If the user took the skip-path, this creates a minimal row that satisfies the gate.
   - Call `const result = await cancelSubscription({ userId: user.id, entitlementId: input.entitlementId })`.
   - Return `{ cancelAtPeriodEnd: result.cancelAtPeriodEnd, currentPeriodEnd: result.currentPeriodEnd }`.

Do NOT add any `revalidatePath` calls — Plan 12-02 decides where revalidation happens, and the client components navigate via `router.push('/dashboard')` on success. Do NOT wrap in try/catch — let errors propagate so the client can display them.

If schema field names differ (e.g., `primaryReasonId` vs `primary_reason_id`), run `grep -n "primaryReason" prisma/schema.prisma` first and use the exact camelCase field name Prisma generated. Same for the CancellationSurveyTag join model field names.
